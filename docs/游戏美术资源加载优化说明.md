# 游戏美术资源加载优化说明

## 当前结论

项目之前看到的约 333MB 压力，主要不是核心代码，而是三类内容叠加：

- `node_modules` 开发依赖约 224MB，只用于本地开发，不应进入网页加载包。
- `.git` 历史约 39MB，只影响仓库体积，不会被手机网页下载。
- 运行资源主要来自 BGM、SFX、地图、角色、怪物、武器、UI 图片。

本轮瘦身保留核心玩法和入口，只清理未引用源图、旧素材表，并把音效从 wav 改为 mp3。以后判断“手机加载压力”时，优先看 `dist/assets`，不要把 `node_modules` 和 `.git` 混进网页加载体积。

## 最省加载的美术制作规则

### 1. 源图和运行图分离

生成图、参考图、2x/4x 源图、候选稿不要放进运行时代码会引用的目录。建议：

- 源文件放到 `art-source/` 或项目外归档。
- 游戏真正使用的图只放到 `src/assets/generated/battle-optimized/`、`src/assets/generated/maps/` 等运行目录。
- 运行目录里只保留最终尺寸，不保留 candidate、source、before-cleanup、2x2 大图。

### 2. 按显示尺寸导出

战斗里显示 48px 的怪物，不要导入 512px 或 1024px 的原图。建议按这个规则导出：

- 普通怪物：最长边 64-96px。
- Boss/精英：最长边 128-192px。
- 子弹：最长边 24-64px。
- 拾取物：最长边 24-48px。
- 图鉴/升级卡图：可以单独用 128-256px，不要直接拿战斗大图硬缩放。

移动端 DPR 高，但不需要所有资产都 4x。像素风更适合用低分辨率图片加 `image-rendering: pixelated` 或 WebGL nearest 采样。

### 3. 地图只用循环瓦片

战斗地图不要做 3072x3072 大图，也不要把大图整体渲染。当前推荐方式：

- 每张地图只用一张四方连续循环瓦片。
- 运行图建议 256px 或 512px。
- 屏幕外只保留一圈安全区，角色移动时平铺补齐。
- 不渲染整张“无限地图”，只渲染当前视口和安全区。

### 4. 图片优先合并和预加载

战斗内频繁出现的资产要提前加载：

- 进入战斗前预加载当前地图、玩家、当前地图会刷的怪物、已有武器子弹、拾取物、经验条框。
- 不要在怪物生成、死亡、升级选择时临时创建图片。
- 同类小图可以做图集，减少请求数量和纹理切换。

### 5. 音频按用途压缩

音频是手机加载压力的大头之一：

- BGM 用 mp3，建议 80-96kbps，循环音乐不要超过必要长度。
- SFX 用 mp3/ogg，短音效建议 32-64kbps。
- 不要把 wav 放进运行包。
- 战斗内高频音效必须聚合播放，不能每次命中/每颗经验都新开一个音频。

### 6. UI 不要每个按钮一张图

按钮、面板、边框优先用 CSS 或 WebGL 九宫格/基础图形绘制：

- 只有复杂装饰框使用图片。
- 普通按钮不要每个状态一张独立大图。
- 字体保持清晰可读，像素风可以靠边框、阴影、色块表达，不必把文字烘焙进图片。

## 新资产进入项目的检查清单

新增美术前先确认：

- 这张图是否会在战斗内显示？如果是，是否已经导出到实际显示尺寸？
- 是否有 source、candidate、参考图混进 `src/assets`？
- 是否能用循环瓦片替代大地图？
- 是否能用图集或预加载减少运行中请求？
- 是否会被 Vite 打进 `dist/assets`？如果会，体积是否合理？
- 手机端是否会在战斗开始、生成怪物、攻击、升级时才首次加载它？

## 推荐体积目标

- `src/assets`：控制在 5-10MB。
- BGM：总量控制在 5-8MB。
- SFX：总量控制在 0.3-1MB。
- `dist/assets`：首版目标控制在 10-15MB。
- 整个源码仓库不含 `node_modules`、`.git`、`dist`：控制在 20MB 左右。

如果要交付或上传运行版本，应该交付 `dist`；如果要交付源码，应该排除 `node_modules`。这两种体积不要混在一起算。
